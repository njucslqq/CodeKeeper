# Issue 深度分析系统 (PRD) 需求文档

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 构建一个基于 LLM 的 Issue 深度分析系统，支持多仓库、多文档系统、多维度分析、可扩展、高并发、服务化部署

**Architecture:**
- Web 服务架构（FastAPI），提供 RESTful API
- 服务启动时加载稳定配置（Git 仓库、文档系统、LLM、降级策略、历史记录）
- 请求参数动态传入（Issue IDs、输出格式、分析维度选择）
- 异步分析 + 并发处理 + WebSocket 进度通知
- Prompt 模板化配置（Markdown 格式，支持自定义扩展）
- 插件系统（目录扫描、独立运行）
- 历史记录管理
- Docker/K8s/本地部署支持，健康检查端点，透明度监控（Prometheus/Telemetry）

**Tech Stack:** FastAPI, Python 3.11+, OpenAI/Anthropic SDK, asyncio, websockets, prometheus_client, uvicorn

---

## 📋 需求清单

### 【基础部分】服务与配置

#### 服务入口
- **1. Web 服务**：系统作为 Web 服务持续运行，基于 FastAPI 框架
- **2. RESTful API**：提供分析服务的 RESTful API 端口

#### 服务配置（稳定，启动时通过 --cfg 传入加载）
- **3. 服务配置文件命名**：建议使用 `ai-analyzer.yaml`
- **4. Git 仓库配置**：Git 仓库类型、仓库列表获取配置
- **5. Git 认证配置**：Git 仓库的认证信息（API Token、用户名密码）
- **6. 文档系统配置**：文档系统类型、接入配置、认证配置
- **7. LLM 配置**：模型选择（OpenAI/Anthropic）、API 配置
- **8. 降级策略配置**：LLM 调用失败时的处理策略
  - 重试次数和退避策略
  - 使用备用模型降级
  - 降级为简单分析（非 LLM）
- **9. 历史记录配置**：保存多久的历史分析记录（如 30 天）
- **10. 服务启动**：通过 `--cfg` 参数传入配置文件路径
- **11. 热重载支持**：暂不考虑热重载

#### 服务健康检查与监控
- **12. 健康检查端点**：提供 `/health` 端点，返回服务状态、依赖服务健康状态
- **13. 服务透明度**：集成 Prometheus/Telemetry，提供以下监控指标：
  - 吞吐量（请求/秒）
  - 负载（并发任务数、队列长度）
  - 性能（平均响应时间、P95/P99 响应时间）
  - 压力（CPU、内存使用率）
  - 状态统计（成功/失败/超时计数）
  - 业务指标（分析完成数、LLM 调用次数、平均分析时长）
- **14. 监控最佳实践**：参考业界最佳实践设计监控指标和采集方式

#### 请求参数（外部 API 调用）
- **15. 批量 Issue 输入**：支持一次分析多个 issue-id
- **16. 输出格式选择**：支持 html、markdown
- **17. 输出方式选择**：合并输出、独立输出、两者都需
- **18. 分析维度和子维度选择**：支持自由选择和组合具体的分析类型

#### 认证与速率限制
- **19. API 认证**：支持 API Key 认证（可选，根据配置启用）
- **20. 速率限制**：防止滥用，可配置每用户/IP 的请求频率限制

#### 异步处理
- **21. 异步分析**：LLM 调用采用异步方式（asyncio）
- **22. 并发控制**：分析任务的并发数可配置，避免资源耗尽

#### WebSocket 进度通知
- **23. WebSocket 端点**：提供 `/ws/progress/{task_id}` 端点
- **24. 实时进度推送**：分析过程中的实时进度推送给客户端
- **25. 任务状态追踪**：每个任务有唯一 ID，支持状态查询

### 【数据获取部分】Git 仓库与文档

#### Git 仓库支持扩展
- **26. Git 仓库类型扩展**：支持多种 Git 仓库类型
  - GitHub（通过 GitHub API）
  - GitLab（通过 GitLab API）
  - 其他自定义 Git 仓库（可扩展）

#### Git 仓库数量支持
- **27. 多仓库支持**：代码仓库可能是一个或多个
  - 如果是多个，遍历所有仓库中跟指定 issue-id 相关的提交
  - 合并所有仓库的提交后做整体分析

#### Git 仓库列表获取扩展
- **28. 仓库列表获取扩展**：支持多种方式获取仓库列表
  - 通过配置文件提供仓库列表
  - 指定一个 superproject 的本地目录
  - 自动根据 git submodule 获取子仓库列表
  - 其他自定义方式（可扩展）

#### Git 提交数据获取
- **29. Issue 提交查询**：从所有 Git 仓库中查询到这个 issue 的所有提交（合并后）
- **30. Commit Message 规范扩展**：commit 消息解析出 issue-ID
  - 支持扩展，不能写死
  - issue-ID 一定会有
  - 规范可能会变，支持自定义解析

#### 文档系统支持扩展
- **31. 文档系统扩展架构**：支持扩展多种文档系统
  - Confluence（通过 REST API）
  - 飞书（通过 API）
  - 其他自定义文档系统（可扩展）

#### 文档标识扩展
- **32. 文档标识扩展**：每个文档都有名称规范，里面可以解析出 issue-ID
  - 支持扩展，不能写死
  - issue-ID 一定会包含在里面
  - 规范可能会变，支持自定义解析

#### 文档类型支持扩展
- **33. 文档类型扩展架构**：支持扩展多种文档类型
  - 需求文档
  - 设计文档
  - 发布文档
  - 运维文档
  - 其他自定义文档类型（可扩展）

#### 文档获取
- **34. 文档获取**：从文档系统中检索到这个 issue 对应的所有关联文档

#### Issue 基础信息支持扩展
- **35. Issue 基础描述扩展**：每个 issue 都会有基础的 issue description
  - 支持 Jira issue（通过 Jira API）
  - 支持 GitHub issue（通过 GitHub API）
  - 支持 GitLab issue（通过 GitLab API）
  - 其他自定义 issue 类型（可扩展）

#### Issue 关联关系抽象设计
- **36. Issue 关联类型定义**：抽象出常见的关联关系类型
  - **related**：关联但不一定有依赖关系
  - **derived**：派生/子 issue
  - **blocked**：阻塞/前置 issue
  - 其他自定义关联类型（可扩展）

#### Issue 关联查询
- **37. Issue 关联查询扩展**：issue 可能会关联到别的 issue，这些信息也需要查询出来
  - 支持 Jira（issue links）
  - 支持 GitHub（related issues）
  - 支持 GitLab（related issues）
  - 其他自定义关联（可扩展）
  - 后续分析可能会用到

#### 数据缺失和边界情况处理
- **38. 数据缺失处理**：对于一个 issue-ID，它的提交数据、文档数据可能不存在
  - 文档数据缺失时：
    - 【60】需求一致性审查 无法执行
    - 【67】设计一致性审查 无法执行
    - 【75】需求一致性审查 无法执行
    - 但【73】反向业务分析 可以继续执行
    - 但【72】代码实现分析 可以继续执行
  - 提交数据缺失时：
    - 所有依赖代码的分析都无法工作
    - 但基于文档的分析（如需求解析）仍可工作
- **39. 影响降到最低**：在数据缺失下的处理原则
  - 不影响的功能应该还能继续 work
  - 影响要降到最低
  - 清晰标记哪些分析因数据缺失而跳过

### 【并发与输出管理】多 Issue 并发分析

#### 多 Issue 并发分析支持
- **40. 批量输入支持**：支持一次给出多个 issue-id 进行分析
- **41.1 并发分析配置**：多个 issue 并发执行（性能和效率优化）
  - 可配置的 issue 间并发数（默认并发度）
  - 可配置的维度内并发数（默认串行）
- **42. 并发状态管理**：在并发情况下要注意多个分析任务状态的管理和透明度的支持
  - 进度实时显示（通过 WebSocket）
  - 每个任务的独立状态（排队、处理中、完成、失败）
  - 任务队列管理（优先级、超时处理）
- **43. 并发日志隔离**：日志不能写乱了，需要线程安全的日志
  - 每个任务的独立日志文件（可选，按配置启用）
  - 统一的结构化日志格式（时间、级别、模块、任务ID、消息）
- **44. 并发输出隔离**：输出的分析文档也不能写乱了，需要文件写入的同步或隔离

#### 多 Issue 输出管理
- **45. 输出文件命名**：多个输出的 html 或 markdown 文件名需要明确
  - 格式：`issue-{issue_id}-{timestamp}.{ext}`
  - 或由配置指定命名模式
- **46. 输出方式选择**：用户可以选择多个 issue 的输出方式
  - **47. 合并输出**：所有 issue 的分析结果合并到一个文件
  - **48. 独立输出**：每个 issue 一个独立文件
  - **49. 两者都需**：既生成独立文件，也生成合并文件
- **50. 输出规则配置**：通过配置文件来自定义输出的规则
  - 支持正则表达式匹配规则
  - 符合规则的 issue 合并到一起输出
  - 不符合的 issue 独立输出
- **51. 输出报告本地存储**：分析报告输出到本地文件系统
  - 输出目录可配置（默认 `./reports`）
  - 支持目录自动创建和清理策略

#### 输出报告模板化
- **52. 报告模板化支持**：支持报告样式模板化
  - 内置模板（简洁、标准、详细）
  - 支持自定义模板（通过配置指定模板文件路径）
  - 模板引擎：Jinja2
  - 模板变量：标准化的上下文数据结构

### 【Prompt 配置与 LLM 分析】Prompt 模板系统

#### Prompt 配置管理
- **53. 默认 Prompt 机制**：默认 Prompt 作为配置文件管理
  - 内置默认 Prompt（硬编码在代码中）
  - 配置文件路径：与 `ai-analyzer.yaml` 同目录下的 `prompts/` 目录
  - Prompt 文件命名：按维度和子维度命名，如 `business/original-requirements.md`
- **54. 自定义 Prompt 配置**：程序支持用户通过配置导入自定义的分析 Prompt
  - 用户可以添加新的 Prompt 文件到配置目录
  - 用户可以覆盖内置 Prompt
- **55. Prompt 格式**：采用 Markdown 格式，支持 Jinja2 模板变量
  - 模板变量示例：`{{issue}}`, `{{requirements}}`, `{{code}}`, `{{design}}`, `{{context}}`
- **56. Prompt 验证机制**：验证 Prompt 文件语法和变量有效性
- **57. Prompt 拼接策略**：如果用户配置了自定义的 Prompt
  - 则在默认 Prompt 的末尾拼接用户自定义 Prompt
  - 或者使用自定义 Prompt 完全替换（可配置）

#### LLM 调用方式支持
- **58. LLM 调用实现扩展**：支持多种实现方式
  - 优先实现 Claude CLI 支持（本地命令行调用）
  - 也支持 OpenAI API 调用
  - 也支持 Anthropic API 调用
- **59. 命令行调用**：通过 subprocess 调用 Claude/CodeX CLI
  - 支持超时配置
  - 支持环境变量传递
  - 支持输出捕获

#### 历史记录管理
- **60. 历史记录保存**：保存分析结果作为历史记录
  - 存储位置：可配置（默认 `./history`）
  - 记录格式：SQLite 数据库（推荐，便于查询）
  - 记录内容：任务 ID、Issue ID、分析维度、分析结果、时间戳、耗时
- **61. 历史记录清理**：根据保存时长配置定期清理过期记录
  - 清理策略：按时间（保留最近 N 条）
  - 或按数量（保留最新 N 条）
- **62. 历史查询 API**：提供 API 查询历史记录
  - 支持分页、过滤（按 issue、按时间、按状态）
  - 支持排序（按时间、按耗时）

### 【核心部分】深度分析（三个维度，支持扩展）

#### 分析维度扩展架构
- **63. 维度可扩展**：三个分析维度（业务、技术、流程）支持扩展
  - 维度注册机制：通过配置或插件注册新维度
  - 维度优先级：可配置分析维度的执行顺序

#### 子维度可扩展
- **64. 子维度可扩展**：每个维度下的分析子维度支持扩展
  - 子维度注册机制：通过配置或插件注册新子维度
  - 子维度依赖关系：支持定义子维度间的依赖和执行顺序

#### 业务维度深度分析（支持扩展）
- **65. 分析原始业务需求**：需求文档中记录的原始业务需求
  - 提取业务目标、痛点、预期效果
  - Prompt 模板：使用 `business/original-requirements.md`
  - LLM 分析：将需求文档发送给 LLM 进行结构化提取
- **66. 反向业务分析**：通过代码实现反向分析出来的业务层面需求
  - 提取实际实现的业务功能、用户价值
  - Prompt 模板：使用 `business/reverse-analysis.md`
  - LLM 分析：将代码提交信息发送给 LLM 进行业务意图分析
- **67. 需求一致性审查**：深度分析【65】和【66】的匹配情况
  - Prompt 模板：使用 `business/requirement-consistency.md`
  - LLM 分析：将需求文档和代码实现摘要发送给 LLM 进行一致性审查
  - 分析内容：
    - 代码实现是否完整
    - 代码实现是否正确实现了需求文档中描述的需求
    - 代码实现是否符合需求文档中的规格要求
    - 实现和需求是否匹配
    - 功能覆盖率
    - 缺失的功能列表
    - 额外的功能列表
  - **审查评估报告输出**：
    - 风险评级（高/中/低）
    - 后续需要优化完善的列表
    - 具体的问题列表

#### 技术维度深度分析（支持扩展）
- **68. 分析原始设计规格**：设计文档中记录的原始设计规格和要求
  - 提取设计约束、技术方案、架构决策
  - Prompt 模板：使用 `technical/original-design.md`
  - LLM 分析：将设计文档发送给 LLM 进行结构化提取
- **69. 代码实现深度分析**：深度分析 issue 的代码实现具体情况
  - Prompt 模板：使用 `technical/code-implementation.md`
  - LLM 分析：将代码提交和文件内容发送给 LLM 进行深度技术分析
  - 分析内容：
    - 多线程问题处理方式
    - 接口设计（RESTful、gRPC、事件驱动等）
    - 关键数据结构（数据模型、存储策略）
    - 核心逻辑和算法
    - 是否符合 SOLID 设计原则
    - 代码复杂度评估
    - 代码可维护性评估
    - 对代码实现的总体评分
    - 性能分析（时间复杂度、空间复杂度）
    - 安全分析（潜在安全风险）
- **70. 设计一致性审查**：深度分析【68】和【69】的匹配情况
  - Prompt 模板：使用 `technical/design-consistency.md`
  - LLM 分析：将设计文档和代码实现摘要发送给 LLM 进行一致性审查
  - 分析内容：
    - 真正的实现是否和设计文档中的要求匹配
    - 是否出现上实现的偏离
    - 设计了但未实现的功能
    - 实现了但未设计的了能
    - 架构一致性评估
    - 偏离风险识别
    - 技术债务识别
  - **审查评估报告输出**：
    - 风险评级（高/中/低）
    - 后续需要优化完善的列表
    - 偏离问题列表
    - 技术债务清单

#### 流程维度深度分析（支持扩展）
- **71. 流程规范检查扩展**：Issue 是否满足研发流程的一些要求和规范
  - Prompt 模板：使用 `process/compliance-check.md`
  - 分析内容：
    - Issue 描述规范检查（是否包含必要信息）
    - 代码提交规范检查（提交消息格式、代码审查状态）
    - 关联文档检查（是否有设计文档、需求文档）
    - 时间节点检查（创建时间、关闭时间、响应时长）
    - 审批流程检查（是否经过必要审批）
  - **流程检查输出**：
    - 风险评估（高/中/低）
    - 后续需要完善的检查点列表
    - 不合规项列表
    - 流程合规度评分

### 【架构部分】系统设计要求

#### FastAPI Web 服务设计
- **72. 框架选择**：FastAPI
  - 选择理由：现代、高性能、原生异步、自动生成 OpenAPI 文档
- **73. 应用结构**：
  - `main.py`：应用入口，FastAPI 实例
  - `api/v1/`：API 路由模块（v1 版本控制）
  - `core/`：核心业务逻辑模块
  - `services/`：外部服务（Git、文档、LLM）
  - `models/`：数据模型（Pydantic）
  - `prompts/`：Prompt 模板目录
  - `config/`：配置管理
  - `tests/`：测试模块

#### API 端点设计
- **74. 分析 API 端点**：`POST /api/v1/analyze`
  - 请求体：包含 Issue IDs、分析配置
  - 响应：任务 ID（用于 WebSocket 追踪）
- **75. 任务查询 API 端点**：`GET /api/v1/tasks/{task_id}`
  - 响应：任务状态、进度、结果
- **76. 历史查询 API 端点**：`GET /api/v1/history`
  - 支持分页、过滤（按 issue、按时间、按状态）
- **77. 健康检查 API 端点**：`GET /health`
  - 响应：服务状态、依赖健康
- **78. Prometheus 指标端点**：`GET /metrics`（如果使用 Prometheus）
  - 或使用 FastAPI 集成的 Prometheus 中间件

#### WebSocket 端点
- **79. WebSocket 端点设计**：`/ws/progress/{task_id}`
  - 支持实时进度推送
  - 支持连接管理

#### 异步任务管理
- **80. 异步任务队列**：使用 `asyncio.Queue` 管理待处理任务
- **81. 任务状态机**：任务状态流转：pending -> processing -> completed/failed
- **82. 任务超时处理**：配置任务最大执行时间，超时自动标记为 failed
- **83. 结果持久化**：任务完成后将结果保存到历史记录
- **84. 并发控制**：使用 `asyncio.Semaphore` 控制并发数
- **85. 任务调度器**：支持优先级队列，优先处理重要任务

#### LLM 异步调用
- **86. 异步 LLM 客户**：支持 OpenAI/Anthropic 的异步 SDK
- **87. 命令行调用**：通过 subprocess 调用 Claude/CodeX CLI
  - 优先级：Claude CLI > CodeX CLI > OpenAI API > Anthropic API
